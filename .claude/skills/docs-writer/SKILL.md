# Kuni-Musubi Docs Writer

## 目的

Kuni-Musubi のドキュメントを、プロダクトビジョン、現在の MVP スコープ、既存ドキュメントの文体と矛盾しない形で作成・更新する。

## 編集前に確認すること

1. 関連するディレクトリの README を先に確認する。
   - `docs/product/README.md`
   - `docs/research/README.md`
   - `docs/technical/README.md`
2. 派生ドキュメントを変更する前に、最も近い一次ドキュメントを読む。
3. 新しいドキュメントを追加した場合は、該当 README を辞書順で更新する。
4. 派生ドキュメントと一次ドキュメントが矛盾する場合は、原則として一次ドキュメントを優先する。一次ドキュメントが古いと判断できる場合は、一次ドキュメントも更新する。

## 文体

- 日本語で書く。
- 簡潔で直接的な文章にする。
- 抽象論よりも、プロダクト上の具体的な意思決定を優先する。
- 見出しはシンプルにする。
- 要件の列挙には箇条書きを使う。
- 比較が読みやすくなる場合だけ表を使う。
- 既存のメタ情報の書き方を維持する。
  - `作成者：泉知成`
  - `作成日：YYYY年M月D日`
  - `更新日：YYYY年M月D日`
- プロダクト名は、既存文言の引用を除き `Kuni-Musubi` を使う。

## 必ず守るプロダクト制約

- MVP ではログイン、アカウント、記事保存、端末間同期を実装しない。
- オンボーディングの回答は任意で、スキップ可能にする。
- ユーザー設定は Cookie ではなく `localStorage` に保存する。
- サーバー側で収集するのは匿名集計イベントのみとする。
- オンボーディング回答と記事行動をユーザー ID で結合しない。
- コメント欄は実装しない。
- 公開いいね、公開カウント、リアクションによるランキングは実装しない。
- `参考になった` は非公開の匿名イベントとして収集してよい。
- MVP では高度な推薦機能を重視しない。
- LLM 処理はユーザーリクエスト時ではなくバッチで実行する。

## README 更新ルール

README のドキュメント一覧を更新するときは、以下を守る。

- 表ではなく箇条書きにする。
- ファイル名の辞書順で並べる。
- 各リンクに、そのドキュメントが何をまとめたものかを 1 行で書く。
- README から見た相対リンクが正しいことを確認する。
