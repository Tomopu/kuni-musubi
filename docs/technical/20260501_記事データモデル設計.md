# 記事データモデル設計

作成者：泉知成  
作成日：2026年5月1日  
更新日：2026年5月9日

## 目的

記事カード表示設計、記事詳細画面表示設計、システムアーキテクチャをもとに、MVP で必要な記事データモデルを定義する。

MVP では、ユーザーアカウント、記事保存、端末間同期は実装しない。そのため、記事データモデルは「記事を収集・解析・表示するためのデータ」と「匿名集計のためのイベント」に分けて設計する。

## 主要エンティティ

MVP で扱う主要エンティティは以下。

- 政党
- 政策カテゴリ
- 記事
- 記事ソース
- 記事表示用コンテンツ
- 記事と政党の関連
- 記事とカテゴリの関連
- 匿名オンボーディングイベント
- 匿名記事イベント
- 日次記事集計
- 日次カテゴリ集計

## テーブル案

### parties

政党の基本情報を保存する。

```txt
id
name
short_name
color_hex
founded_year
leader_name
house_of_representatives_seats
house_of_councillors_seats
ideology_summary
manifesto_summary
manifesto_promises
main_policy_categories
official_url
is_active
display_order
created_at
updated_at
```

`color_hex` は、ホーム画面・政党一覧・オンボーディング・設定画面で使う政党テーマカラーを保存する。

`main_policy_categories` は、政党一覧や政党詳細画面で表示する「主な政党タグ」として使う。記事側の `policy_categories` とは別に、政党そのものの主要政策分野を表す。

`manifesto_promises` は、政党詳細画面で政策・公約を箇条書き表示するための配列として保存する。

### policy_categories

記事に付与するカテゴリを保存する。

```txt
id
name
slug
description
display_order
created_at
updated_at
```

カテゴリ例:

- 税金
- 国防
- 金融政策
- 労働
- 社会保障
- 物価高
- 教育
- 移民
- 外交

### articles

収集した記事・一次情報の基本情報を保存する。

```txt
id
original_title
source_type
primary_source_url
published_at
collected_at
raw_content
important_rank
is_published
status
created_at
updated_at
```

`raw_content` は、スクレイピングで取得した元記事の本文テキストを保存する。重複投稿防止のため `primary_source_url` にはユニーク制約を設けている。

`source_type` の例:

```txt
party_official
government
local_government
news_media
```

`status` の例:

```txt
draft
processed
published
rejected
```

### article_sources

記事に紐づく出典を保存する。

```txt
id
article_id
source_name
source_url
source_type
published_at
created_at
updated_at
```

一次情報と補助的なニュース記事を分けて保存する。

### article_display_contents

ホームの記事カードと記事詳細画面で表示する、加工済みコンテンツを保存する。

```txt
id
article_id
display_title
card_summary
positive_point
life_impact
remaining_issues
public_reactions_summary
thumbnail_type
thumbnail_text
thumbnail_url
created_by
created_at
updated_at
```

`card_summary` は 100 文字程度を目安とする。

`public_reactions_summary` は世論・与野党からの評価を Markdown の箇条書き文字列として保存する。

`thumbnail_type` の例:

```txt
none
text
manual_image
category_default
```

サムネイルは MVP では必須にしない。使う場合は、人間が設定したテキストベースサムネイルまたはカテゴリ別デフォルトを基本とする。

### article_parties

記事と政党の関連を保存する。

```txt
article_id
party_id
relation_type
created_at
```

`relation_type` の例:

```txt
primary
mentioned
opposition_view
```

### article_categories

記事と政策カテゴリの関連を保存する。

```txt
article_id
category_id
display_order
created_at
```

記事カードではカテゴリを最大 2 個まで表示し、3 個以上ある場合は「他2」のように表示する。

### onboarding_events

初回オンボーディングの匿名集計イベントを保存する。

```txt
id
age_group
selected_party_id
selected_party_status
interest_category_ids
created_at
```

`selected_party_status` の例:

```txt
selected
none
unknown
skipped
```

このイベントは個人を識別する ID と結合しない。

### article_events

記事表示・クリック・詳細閲覧などの匿名イベントを保存する。

```txt
id
article_id
event_type
surface
created_at
```

`event_type` の例:

```txt
impression
card_click
detail_view
source_click
helpful_click
unhelpful_click
```

`surface` の例:

```txt
home
party_tab
party_detail
article_detail
```

政治思想と閲覧履歴を個人単位で結合しないため、ユーザー ID は持たない。

`helpful_click` は、記事詳細画面の「参考になった」ボタンが押されたことを表す。

`unhelpful_click` は、記事詳細画面の「そう思わなかった」ボタンが押されたことを表す。

このイベントは公開いいねではなく、匿名・非公開のプロダクト改善用データとして扱う。

### daily_article_stats

記事ごとの日次集計を保存する。

```txt
article_id
date
impressions
card_clicks
detail_views
source_clicks
helpful_clicks
unhelpful_clicks
created_at
updated_at
```

`helpful_clicks` と `unhelpful_clicks` は、どのニュース記事が参考になった、または参考になりづらかったと感じられたかを分析するために利用する。

公開ランキングや政党別人気投票には使わない。

`article_events` の保存時に同日の集計行を upsert し、該当するカウントをインクリメントする。

### daily_category_stats

政策カテゴリごとの日次集計を保存する。

```txt
category_id
date
impressions
card_clicks
detail_views
helpful_clicks
created_at
updated_at
```

カテゴリ単位で、どのテーマの記事が参考になったと感じられたかを把握する。

## 採用しないモデル

MVP では以下を採用しない。

```txt
users
user_profiles
saved_articles
user_article_events
recommendation_logs
```

## 表示データとの対応

### 記事カード

記事カードで使うデータ:

- `article_display_contents.display_title`
- `article_display_contents.card_summary`
- `article_display_contents.thumbnail_type`
- `article_display_contents.thumbnail_text`
- `article_display_contents.thumbnail_url`
- `article_parties`
- `parties.short_name`
- `article_categories`
- `policy_categories.name`
- `articles.published_at`

### 記事詳細画面

記事詳細画面で使うデータ:

- `article_display_contents.display_title`
- `article_display_contents.positive_point`
- `article_display_contents.life_impact`
- `article_display_contents.remaining_issues`
- `article_display_contents.public_reactions_summary`
- `article_sources`
- `article_parties`
- `article_categories`
- `article_events`

記事詳細画面の「参考になった」「そう思わなかった」は `POST /analytics/article-event` で `helpful_click` / `unhelpful_click` として送信する。反応は匿名・非公開で、画面上に件数は表示しない。

記事詳細画面の共有ボタン（X、LINE、リンクコピー）はブラウザ上で共有 URL を生成する。現時点では DB に保存しない。

## 理解支援用データ

MVP では必須にしないが、将来的に記事詳細画面で「ふりがな」や「説明をもっと詳しく」を実装する場合は、以下のようなデータを追加する。

### article_explainers

```txt
id
article_id
target_text
ruby
short_explanation
detailed_explanation
display_order
created_at
updated_at
```

用途:

- 難しい政治用語にふりがなを付ける。
- 「説明をもっと詳しく」で表示する補足説明を保存する。

リアルタイムで LLM を呼び出すのではなく、バッチ処理で事前生成し、必要に応じて人間が確認・修正する。
