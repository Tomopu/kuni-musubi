# バッチ処理運用ガイド

作成者：泉知成
作成日：2026年5月6日
更新日：2026年5月6日

---

## 概要

`batch/` ディレクトリには、政党公式サイトの RSS フィードを取得し、Gemini API で記事を整形・要約して DB に保存するバッチ処理が実装されている。

本ドキュメントは、バッチ処理の操作方法、セーフティーガードの仕様、環境変数の設定方法をまとめる。

---

## セットアップ

### 必要な環境変数

`.env` ファイルまたはシェル環境変数として設定する。

| 変数名 | 必須 | デフォルト値 | 説明 |
|---|---|---|---|
| `DATABASE_URL` | Yes | `postgresql://kuni_musubi:kuni_musubi@localhost:5432/kuni_musubi` | PostgreSQL 接続文字列 |
| `GEMINI_API_KEY` | Yes（`--dry-run` 以外） | `""` | Gemini API キー |
| `LLM_MODEL` | No | `gemini-2.0-flash` | 使用する Gemini モデル名 |
| `SCRAPER_TIMEOUT` | No | `15` | HTTP タイムアウト（秒） |
| `RSS_MAX_ITEMS` | No | `50` | RSS フィードあたりの最大取得件数 |
| `SCRAPER_REQUEST_DELAY_SECONDS` | No | `3.0` | リクエスト間の最小待機秒数 |
| `SCRAPER_DOMAIN_INTERVAL_SECONDS` | No | `10.0` | 同一ドメインへの連続アクセス間隔（秒） |
| `SCRAPER_MAX_ITEMS_PER_RUN` | No | `30` | 1バッチ実行あたりの最大処理記事数 |
| `SCRAPER_MAX_RETRIES` | No | `1` | HTTP エラー時の最大リトライ回数 |
| `SCRAPER_BACKOFF_FACTOR` | No | `2.0` | 指数バックオフの乗数 |

デフォルト値はすべて「相手サーバーに負荷をかけない保守的な設定」になっている。本番運用でも基本的にデフォルトのまま使用することを推奨する。

### 実行前の確認

```bash
# backend の DB モデルを共用するため PYTHONPATH に backend を追加する
export PYTHONPATH=../backend

# 依存パッケージのインストール
cd batch
uv sync
```

## CLIコマンドリファレンス

バッチ処理のエントリポイントは `batch/main.py` である。

### 基本構文

```bash
# batch/ ディレクトリから実行する
cd /path/to/kuni-musubi/batch
PYTHONPATH=..:../backend uv run python -m batch.main [オプション]
```

---

### `--fetch-only`（スクレイピング動作確認モード）

**用途**: RSS フィード取得のみ行い、LLM 処理も DB 保存も一切行わない。政治系サイトへのアクセス動作を最小限のコストで確認するときに使用する。

**実行コマンド**:
```bash
PYTHONPATH=..:../backend uv run python -m batch.main --fetch-only
```

**出力例**:
```
[pipeline] フェッチ完了: 30 件
[fetch-only]   1. [自由民主党] 少子化対策の新制度が閣議決定
             URL: https://www.jimin.jp/news/press/123456.html
             公開日: 2026-05-06T10:00:00+09:00
             本文長: 842 文字
[fetch-only]   2. [立憲民主党] ...
...
[pipeline] fetch-only 完了: 30 件
```

**特徴**:
- GEMINI_API_KEY が不要
- DB 接続が不要
- セーフティーガード（レートリミット・robots.txt チェック）は通常どおり動作する

---

### `--dry-run`（LLM 動作確認モード）

**用途**: RSS 取得から LLM による要約生成まで行うが、DB への書き込みはスキップする。LLM のプロンプト調整時や出力品質の確認に使用する。

**実行コマンド**:
```bash
PYTHONPATH=..:../backend uv run python -m batch.main --dry-run
```

**出力例**:
```
[pipeline] フェッチ完了: 30 件
[pipeline] dry_run: 少子化支援の新制度、2026年度に本格スタート
[pipeline] dry_run: 中小企業への補助金拡充、申請受付を開始
...
[pipeline] 完了: fetch=30, processed=28, saved=0, errors=2
```

**注意**:
- GEMINI_API_KEY が必要（Gemini API を呼び出す）
- DB 接続は不要（保存処理をスキップするため）

---

### 通常実行（引数なし）

**用途**: 本番運用。RSS 取得 → LLM 処理 → DB 保存の全ステップを実行する。

**実行コマンド**:
```bash
PYTHONPATH=..:../backend uv run python -m batch.main
```

**出力例**:
```
[pipeline] フェッチ完了: 30 件
[pipeline] 上限適用: 30 → 30 件
[pipeline] 保存完了: a1b2c3d4-...
[pipeline] 保存完了: e5f6g7h8-...
...
[pipeline] 完了: fetch=30, processed=28, saved=28, errors=2
```

## セーフティーガードの仕様

政治系サイトへの負荷を最小限に抑えるため、以下のガード機構が実装されている。

### 1. robots.txt チェック

各フィード URL および記事 URL にアクセスする前に、そのドメインの `robots.txt` を取得して `Kuni-Musubi-Bot/1.0` に対するアクセス許可を確認する。

- アクセス禁止の場合: `PermissionError` を送出してスキップ
- `robots.txt` が取得できない場合: フェイルオープン（アクセス許可として扱う）

### 2. リクエスト間の強制待機

各 HTTP リクエストが成功した後、次のリクエストまで `SCRAPER_REQUEST_DELAY_SECONDS`（デフォルト 3.0 秒）必ず待機する。

### 3. 同一ドメインのレートリミット

`DomainRateLimiter` がドメインごとの最終アクセス時刻を管理し、同一ドメインへのアクセス間隔が `SCRAPER_DOMAIN_INTERVAL_SECONDS`（デフォルト 10.0 秒）以上になるよう制御する。

複数フィードを持つサイト（例: 同じ政党サイトの複数 URL）でも適切な間隔が保たれる。

### 4. 指数バックオフ付きリトライ

`429 Too Many Requests` および `503 Service Unavailable` を受信した場合は**即時停止**してリトライしない。

その他の HTTP エラー（500 系など）は `SCRAPER_MAX_RETRIES`（デフォルト 1）回まで、以下の式でバックオフ待機してからリトライする。

```
待機秒数 = SCRAPER_REQUEST_DELAY_SECONDS × SCRAPER_BACKOFF_FACTOR^(attempt - 1)
```

例: `DELAY=3.0`, `FACTOR=2.0` のとき、1 回目のリトライは 3.0 秒待機、2 回目は 6.0 秒待機。

### 5. 1バッチあたりの最大処理件数

`SCRAPER_MAX_ITEMS_PER_RUN`（デフォルト 30）を超える記事が取得された場合は先頭 30 件のみを LLM 処理・保存する。

`--fetch-only` モードでは制限を適用せずに全件を表示する（取得のみで副作用がないため）。

---

## エラー発生時の挙動

| エラーの種類 | 挙動 |
|---|---|
| robots.txt でアクセス禁止 | そのフィードをスキップして次へ |
| HTTP 429 / 503 | そのフィードをスキップして次へ（リトライなし） |
| HTTP 500 系 | `SCRAPER_MAX_RETRIES` 回リトライ後にスキップ |
| LLM 処理失敗 | その記事をスキップして次の記事へ |
| DB 保存失敗 | ロールバックしてその記事をスキップ |

エラーは `PipelineResult.errors` に蓄積され、実行完了後に一括表示される。

## テスト

セーフティーガードのユニットテストは `batch/tests/scrapers/test_safety.py` に実装されている。

```bash
cd batch
uv run pytest tests/scrapers/test_safety.py -v
```

テストでは実際の HTTP リクエストを発生させないよう、`unittest.mock` でネットワーク呼び出しをモック化している。
