# ソフトウェアアーキテクチャ

作成者：泉知成  
作成日：2026年5月1日  
更新日：2026年5月9日

## 目的

Kuni-Musubi のアプリケーション内部の責務分割と、ファイル・ディレクトリの命名規則を定義する。

システムアーキテクチャは、Next.js、FastAPI、Python バッチ、PostgreSQL などの構成要素がどう接続されるかを扱う。一方で、ソフトウェアアーキテクチャは、それぞれのアプリケーション内部をどのように分割し、どの方向に依存させるかを扱う。

## 全体方針

MVP では、過度に重い Clean Architecture や DDD を導入しない。

ただし、FastAPI の route に DB 処理や複雑な業務処理を直接書くと、後から修正しづらくなる。そのため、薄いレイヤード構成を採用する。

方針:

- Frontend: Feature-based Architecture
- Backend: Layered Architecture + Usecase
- Batch: Pipeline Architecture

## Backend

### 基本構成

```txt
backend/
  app/
    main.py
    api/
      articles.py
      parties.py
      categories.py
      analytics.py
      admin/
        router.py
        auth.py
    usecases/
      list_articles.py
      get_article_detail.py
      list_parties.py
      get_party_detail.py
      record_onboarding_event.py
      record_article_event.py
    domain/
      article.py
      party.py
      category.py
      analytics.py
    infrastructure/
      db/
        session.py
        models.py
        repositories/
          article_repository.py
          party_repository.py
          category_repository.py
          analytics_repository.py
    schemas/
      article.py
      party.py
      category.py
      analytics.py
```

### 責務

#### api/

FastAPI の route を置く。

責務:

- HTTP request を受け取る
- Pydantic schema で request / response を扱う
- usecase を呼び出す
- HTTP status code を返す

`api/` には DB クエリや LLM 処理を書かない。

管理画面ルートは `api/admin/` に置く。Cookie ベースのセッション認証で保護し、Jinja2 テンプレートで HTML を返す。管理画面は FastAPI の通常 API とは異なりサーバーレンダリングで動作する。

#### usecases/

アプリケーションの処理手順を置く。

責務:

- 記事一覧を取得する
- 記事詳細を取得する
- 政党一覧を取得する
- 匿名オンボーディングイベントを記録する
- 匿名記事イベントを記録する

`usecases/` は、API と DB の間に置く薄いアプリケーション層である。

#### domain/

Kuni-Musubi 固有の概念やルールを置く。

責務:

- Article, Party, Category などの概念を表現する
- 記事カードに表示するカテゴリ数の上限
- 公開いいねを持たない方針
- 匿名イベントの扱い
- 記事表示に関する簡単な判定

MVP では `domain/` は小さく始める。将来的にロジックが増えた場合のみ拡張する。

#### infrastructure/

DB 接続、SQLAlchemy モデル、Repository、外部サービス接続などの技術的詳細を置く。

責務:

- DB セッション管理
- ORM モデル定義
- DB クエリ
- 永続化処理
- 外部 API クライアント
- インフラ依存の実装

#### schemas/

FastAPI の Pydantic schema を置く。

責務:

- request schema
- response schema
- API 入出力の型定義

## Backend の依存方向

依存方向は以下を基本とする。

```txt
api → usecases → infrastructure/db/repositories
          ↓
        domain
```

ルール:

- `api/` は `usecases/` を呼ぶ。
- `api/` から `infrastructure/db/repositories/` を直接呼ばない。
- `usecases/` は `infrastructure/db/repositories/` を使う。
- `domain/` は FastAPI や SQLAlchemy に依存しない。
- `infrastructure/` は DB や外部 API などの技術的詳細を担当する。
- LLM 処理は API では実行しない。

## Frontend

### 基本構成

```txt
frontend/
  src/
    app/
      onboarding/
      articles/
        [articleId]/
      parties/
        [partyId]/
      settings/
      about/
    features/
      onboarding/
      articles/
      parties/
      settings/
      analytics/
    components/
      ui/
      layout/
    lib/
      api/
      storage/
      constants/
      utils/
```

### 責務

#### app/

Next.js App Router の route を置く。

責務:

- ページ単位の entry point
- layout
- metadata
- routing

#### features/

機能単位の UI・hooks・状態管理を置く。

例:

- `features/articles/`
  - 記事カード
  - 記事詳細
  - 記事一覧取得 hooks

- `features/onboarding/`
  - 初回オンボーディングフォーム
  - localStorage 保存処理
  - 匿名イベント送信

- `features/parties/`
  - 政党一覧
  - 政党詳細

#### components/

汎用 UI コンポーネントを置く。

例:

- Button
- Tag
- Card
- Modal
- Layout
- Header
- Footer

#### lib/

API クライアント、localStorage 操作、定数、ユーティリティを置く。

重要:

- localStorage の読み書きは UI コンポーネントに直接書かない。
- `lib/storage/` に閉じ込める。
- API fetch 処理は `lib/api/` に閉じ込める。

## Batch

### 基本構成

```txt
batch/
  pipelines/
    article_pipeline.py
  steps/
    fetch.py
    llm_process.py
    save.py
  llm/
    prompts/
      system_prompt.txt
      user_prompt.txt
      article.py
    schemas/
      article.py
  scrapers/
    html_parser.py
    html_index_scraper.py
    safety.py
  config/
    parties.yml
  main.py
```

### 責務

#### pipelines/

一連のバッチ処理の流れを定義する。

現在の実装:
- `article_pipeline.py`: RSS/HTML から記事を取得し、LLM で処理して DB に保存する単一のパイプライン。`--fetch-only`, `--dry-run`, `--url` オプションで動作を制御できる。

#### steps/

パイプライン内の個別ステップを置く。

- `fetch.py`: RSS フィードおよび HTML インデックスから記事を取得する
- `llm_process.py`: Gemini API で記事テキストを処理し、ArticleLLMOutput を返す
- `save.py`: LLM 出力を DB の各テーブルに保存する

#### llm/

LLM に関するプロンプト、出力 schema、バリデーションを置く。

LLM は表示時に呼ばず、バッチで事前実行する。

#### scrapers/

スクレイパーユーティリティを置く。

- `html_parser.py`: 記事本文テキストを HTML からスクレイピングする
- `html_index_scraper.py`: CSS セレクタを使って政党サイトの記事 URL 一覧を収集する
- `safety.py`: robots.txt チェック、ドメインレートリミット、バックオフ計算を担う

## 命名規則

### 全般

- ディレクトリ名は `snake_case` を基本とする。
- Python ファイル名は `snake_case.py` とする。
- TypeScript ファイル名は、用途に応じて `kebab-case` または `PascalCase` を使い分ける。
- DB テーブル名は `snake_case` の複数形とする。
- API の path は `kebab-case` とする。

## MVP でやりすぎないこと

- `application/usecases` のように意味の薄い二段階構造を作らない。
- `domain/models` しかないなら、`domain/` 直下に置く。
- 本格的な DDD の集約・イベント・ポートを最初から作らない。
- API route に DB 処理を直接書かない。
- LLM 処理を API request のたびに実行しない。
- ユーザーアカウントや保存記事を前提にした構造を作らない。

## 重要な境界

- 表示用記事データと元記事データを分ける。
- LLM 生成ロジックを API から切り離す。
- 匿名分析イベントをユーザー設定と結合しない。
- スクレイパーを政党ごとに差し替えられるようにする。
- localStorage 処理を UI から分離する。
- 「参考になった」は公開いいねではなく、匿名・非公開イベントとして扱う。
