# DBテーブルまとめ

作成者：泉知成  
作成日：2026年5月8日  
更新日：2026年5月8日

## 目的

現在実装されている PostgreSQL のテーブルと、各カラムに保存されるデータをまとめる。

本ドキュメントは設計案ではなく、`backend/app/infrastructure/db/models.py` の SQLAlchemy モデルを基準にする。

## 全体像

現在の実装では、以下の 12 テーブルを使う。

| テーブル | 主な用途 |
|---|---|
| `parties` | 政党の基本情報、議席数、テーマカラー、政策情報を保存する |
| `policy_categories` | 記事に付与する政策カテゴリを保存する |
| `articles` | 収集した記事の基本情報、公開状態を保存する |
| `article_display_contents` | LLM が生成した記事カード・記事詳細用の表示文言を保存する |
| `article_sources` | 記事の出典・一次情報 URL を保存する |
| `article_parties` | 記事と政党の関連を保存する中間テーブル |
| `article_categories` | 記事と政策カテゴリの関連を保存する中間テーブル |
| `onboarding_events` | 初回オンボーディングの匿名イベントを保存する |
| `article_events` | 記事表示・クリック・リアクションの匿名イベントを保存する |
| `daily_article_stats` | 記事ごとの日次集計を保存する |
| `daily_category_stats` | 政策カテゴリごとの日次集計を保存する |
| `admin_users` | 管理画面にログインする管理者ユーザーを保存する |

## `parties`

政党一覧、政党詳細、オンボーディング、設定画面、記事カード内の政党表示で使う政党マスタ。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | 政党 ID |
| `name` | VARCHAR(100) | 政党の正式名称。例: `自由民主党` |
| `short_name` | VARCHAR(20) | 画面表示用の短縮名。例: `自民` |
| `color_hex` | VARCHAR(7) | 政党テーマカラー。例: `#e60012` |
| `is_active` | BOOLEAN | 一覧/APIに表示する政党かどうか |
| `display_order` | INTEGER | 政党一覧での表示順 |
| `founded_year` | INTEGER | 結成年 |
| `leader_name` | VARCHAR(100) | 代表者名 |
| `house_of_representatives_seats` | INTEGER | 衆議院の議席数 |
| `house_of_councillors_seats` | INTEGER | 参議院の議席数 |
| `ideology_summary` | TEXT | 党の政治理念の要約 |
| `manifesto_summary` | TEXT | 政策・公約の要約 |
| `manifesto_promises` | TEXT[] | 政党詳細で箇条書き表示する主要政策・公約 |
| `main_policy_categories` | TEXT[] | 政党一覧/詳細で表示する主な政党タグ |
| `official_url` | VARCHAR(2048) | 公式サイト URL |
| `created_at` | TIMESTAMPTZ | レコード作成日時 |
| `updated_at` | TIMESTAMPTZ | レコード更新日時 |

`main_policy_categories` は記事カテゴリとは別物で、政党自体の主要政策分野を表す。

## `policy_categories`

記事に付与する政策カテゴリのマスタ。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | 政策カテゴリ ID |
| `name` | VARCHAR(100) | 表示名。例: `税金`, `教育` |
| `slug` | VARCHAR(100) | URL/API向けの一意な識別子 |
| `description` | TEXT | カテゴリ説明 |
| `display_order` | INTEGER | カテゴリ一覧での表示順 |
| `created_at` | TIMESTAMPTZ | レコード作成日時 |
| `updated_at` | TIMESTAMPTZ | レコード更新日時 |

`slug` はユニーク制約を持つ。

## `articles`

バッチ処理で収集した記事の基本情報を保存する。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | 記事 ID |
| `original_title` | VARCHAR(500) | RSS/HTMLから取得した元記事タイトル |
| `source_type` | VARCHAR(50) | 出典種別。例: `party_official`, `demo` |
| `primary_source_url` | TEXT | 元記事または一次情報の URL |
| `published_at` | TIMESTAMPTZ | 元記事の公開日時 |
| `collected_at` | TIMESTAMPTZ | バッチが記事を収集した日時 |
| `status` | VARCHAR(20) | 処理状態。例: `draft`, `processed` |
| `important_rank` | INTEGER | 重要順表示用の順位。未設定なら `NULL` |
| `is_published` | BOOLEAN | APIで公開対象にするかどうか |
| `created_at` | TIMESTAMPTZ | レコード作成日時 |
| `updated_at` | TIMESTAMPTZ | レコード更新日時 |

通常バッチでは、LLM の `quality_flags.needs_human_review` が `false` の場合に `status = processed`, `is_published = true` として保存する。確認が必要な記事は `draft`, `is_published = false` になる。

## `article_display_contents`

記事カードと記事詳細画面に表示する加工済みテキストを保存する。主に LLM 出力から作成される。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | 表示コンテンツ ID |
| `article_id` | UUID | 対象記事 ID |
| `display_title` | VARCHAR(300) | 画面表示用タイトル |
| `card_summary` | TEXT | 記事カード用の短い要約 |
| `thumbnail_type` | VARCHAR(50) | サムネイル種別。例: `none`, `text`, `manual_image`, `category_default` |
| `thumbnail_text` | TEXT | テキストサムネイルに表示する文言 |
| `thumbnail_url` | TEXT | サムネイル画像 URL |
| `thumbnail_alt_text` | TEXT | サムネイル画像の代替テキスト |
| `positive_point` | TEXT | 「何が Good News なのか？」に表示する本文 |
| `life_impact` | TEXT | 「我々の生活への影響」に表示する本文 |
| `remaining_issues` | TEXT | 「残る課題」。JSON文字列化した配列 |
| `public_reactions_summary` | TEXT | 世論・与野党からの評価。JSON文字列 |
| `created_by` | VARCHAR(50) | 作成元。例: `llm_batch` |
| `created_at` | TIMESTAMPTZ | レコード作成日時 |
| `updated_at` | TIMESTAMPTZ | レコード更新日時 |

`article_id` はユニークで、1記事につき表示コンテンツは1件。

## `article_sources`

記事の出典・一次情報を保存する。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | 出典 ID |
| `article_id` | UUID | 対象記事 ID |
| `source_name` | TEXT | 出典名。例: 政党名、媒体名 |
| `source_url` | TEXT | 出典 URL |
| `source_type` | VARCHAR(50) | 出典種別。例: `party_official`, `government`, `demo` |
| `published_at` | TIMESTAMPTZ | 出典側の公開日時 |
| `retrieved_at` | TIMESTAMPTZ | バッチが出典情報を取得した日時 |
| `created_at` | TIMESTAMPTZ | レコード作成日時 |

記事詳細画面の「出典・一次情報」はこのテーブルを使う。

## `article_parties`

記事と政党の多対多関連を保存する中間テーブル。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `article_id` | UUID | 記事 ID |
| `party_id` | UUID | 政党 ID |
| `relation_type` | VARCHAR(20) | 記事と政党の関係。例: `primary`, `mentioned`, `opposition_view` |
| `created_at` | TIMESTAMPTZ | 関連作成日時 |

主キーは `article_id` と `party_id` の複合キー。

バッチ保存時は、LLM が返した政党名を `parties.name` または `parties.short_name` に完全一致で名寄せする。誤紐付けを避けるため、部分一致は使わない。

## `article_categories`

記事と政策カテゴリの多対多関連を保存する中間テーブル。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `article_id` | UUID | 記事 ID |
| `category_id` | UUID | 政策カテゴリ ID |
| `display_order` | INTEGER | 記事内でのカテゴリ表示順 |
| `created_at` | TIMESTAMPTZ | 関連作成日時 |

主キーは `article_id` と `category_id` の複合キー。

バッチ保存時は、LLM が返したカテゴリ名を `policy_categories.name` に完全一致で名寄せする。

## `onboarding_events`

初回オンボーディング完了時の匿名イベントを保存する。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | イベント ID |
| `age_group` | VARCHAR(20) | 選択された年代。例: `10s`, `20s`, `70s_plus` |
| `selected_party_id` | UUID | 選択された支持政党 ID。未選択なら `NULL` |
| `selected_party_status` | VARCHAR(20) | 支持政党選択状態。`selected`, `none`, `unknown`, `skipped` |
| `interest_category_ids` | TEXT[] | 興味のある政策カテゴリ ID の配列 |
| `created_at` | TIMESTAMPTZ | イベント送信日時 |

個人を識別するユーザー ID は保存しない。

## `article_events`

記事表示、クリック、記事詳細閲覧、出典クリック、リアクションを匿名イベントとして保存する。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | イベント ID |
| `event_type` | VARCHAR(50) | イベント種別 |
| `article_id` | UUID | 対象記事 ID |
| `surface` | VARCHAR(50) | イベントが発生した画面 |
| `created_at` | TIMESTAMPTZ | イベント発生日時 |

`event_type` の値:

```txt
impression
card_click
detail_view
source_click
helpful_click
unhelpful_click
```

`surface` の値:

```txt
home
party_tab
party_detail
article_detail
```

`helpful_click` は「参考になった」、`unhelpful_click` は「そう思わなかった」を表す。どちらも公開リアクションではなく、匿名・非公開のプロダクト改善用データ。

## `daily_article_stats`

`article_events` を記事単位・日次で集計するテーブル。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | 集計行 ID |
| `article_id` | UUID | 対象記事 ID |
| `stat_date` | DATE | 集計対象日 |
| `impression_count` | INTEGER | 記事カード表示回数 |
| `card_click_count` | INTEGER | 記事カードクリック回数 |
| `detail_view_count` | INTEGER | 記事詳細閲覧回数 |
| `source_click_count` | INTEGER | 出典リンククリック回数 |
| `helpful_click_count` | INTEGER | 「参考になった」クリック回数 |
| `unhelpful_click_count` | INTEGER | 「そう思わなかった」クリック回数 |
| `created_at` | TIMESTAMPTZ | 集計行作成日時 |
| `updated_at` | TIMESTAMPTZ | 集計行更新日時 |

`article_id` と `stat_date` の組み合わせはユニーク。

`POST /analytics/article-event` でイベントを保存するとき、同じトランザクション内で当日分の集計も upsert する。

## `daily_category_stats`

政策カテゴリ単位の日次集計を保存するテーブル。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | 集計行 ID |
| `category_id` | UUID | 政策カテゴリ ID |
| `stat_date` | DATE | 集計対象日 |
| `helpful_click_count` | INTEGER | そのカテゴリの記事に対する「参考になった」回数 |
| `card_click_count` | INTEGER | そのカテゴリの記事カードクリック回数 |
| `created_at` | TIMESTAMPTZ | 集計行作成日時 |
| `updated_at` | TIMESTAMPTZ | 集計行更新日時 |

現時点の実装では、記事イベント送信時に自動更新しているのは `daily_article_stats` であり、`daily_category_stats` は将来のカテゴリ別分析用の受け皿。

## `admin_users`

管理画面にログインする管理者ユーザーを保存する。

| カラム | 型 | 保存されるデータ |
|---|---|---|
| `id` | UUID | 管理者ユーザー ID |
| `username` | VARCHAR(100) | ログインユーザー名 |
| `hashed_password` | TEXT | bcrypt でハッシュ化したパスワード |
| `created_at` | TIMESTAMPTZ | レコード作成日時 |
| `updated_at` | TIMESTAMPTZ | レコード更新日時 |

`username` はユニーク制約を持つ。管理者ユーザーは `backend/scripts/create_admin.py` で作成・更新する。

## 保存フロー

### バッチ通常実行

`batch.main` の通常実行では、RSS/HTML から取得した記事を Gemini で整形し、以下に保存する。

```txt
articles
article_display_contents
article_sources
article_parties
article_categories
```

`--dry-run` では LLM JSON を検証するが DB には保存しない。

### フロントエンドからの匿名イベント

初回オンボーディング完了時:

```txt
onboarding_events
```

記事表示・クリック・リアクション時:

```txt
article_events
daily_article_stats
```

## 実装上の注意

- 本番のマイグレーション管理はまだ Alembic ではなく、ローカル開発用に `ensure_dev_schema()` が既存 DB ボリュームへ不足カラムを補う。
- `article_display_contents.remaining_issues` と `public_reactions_summary` は DB 型としては `TEXT` で、JSON文字列として保存している。
- ユーザーアカウント、個人 ID、端末 ID、保存記事、個人別閲覧履歴は保存しない。
