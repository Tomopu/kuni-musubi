# セキュリティ設計

作成者：泉知成  
作成日：2026年5月1日  
更新日：2026年5月1日

## 目的

Kuni-Musubi の MVP 開発・運用において考慮すべきセキュリティ方針をまとめる。

本プロダクトでは、ユーザーアカウント、ログイン、記事保存、端末間同期を実装しないため、認証情報や個人プロフィールを扱うリスクは小さい。一方で、政治的嗜好に関する設定、匿名集計イベント、外部サイトのスクレイピング、LLM 生成、パッケージ依存関係など、注意すべき攻撃面は存在する。

## 基本方針

- ユーザーアカウントを作らない。
- 氏名、メールアドレス、電話番号、住所を取得しない。
- 政治的嗜好をアカウントに紐づけない。
- localStorage は表示設定の保存に限定する。
- DB には匿名集計イベントのみ保存する。
- 政治思想と閲覧履歴を個人単位で結合しない。
- 外部入力はすべて信用しない。
- 依存パッケージの取得経路を管理する。
- LLM 出力をそのまま信頼しない。

## サプライチェーン攻撃対策

### 方針

Next.js / TypeScript / Python などの依存パッケージは、サプライチェーン攻撃の主要な侵入口になり得る。

そのため、npm / pnpm / pip / uv などのパッケージ管理ツールが直接外部レジストリへアクセスするのではなく、**Takumi Guard をプロキシとして利用する** 方針とする。

### Takumi Guard の役割

Takumi Guard には以下の役割を期待する。

- パッケージ取得のプロキシ
- 依存パッケージの取得経路の一元化
- 危険なパッケージや不審なバージョンの検知
- 依存関係の監査
- レジストリから取得するパッケージのポリシー制御
- latest version を即時取得しないための遅延制御

### latest version の遅延取得

依存パッケージの最新バージョンは、公開直後に悪意あるコードが混入していても検知が追いつかない可能性がある。

そのため、Takumi Guard 側で **latest version の取得に約3日間のディレイを設ける**。

方針:

- 公開直後の latest version は自動採用しない。
- 原則として、公開から約3日経過したバージョンのみ取得可能にする。
- 緊急のセキュリティパッチについては、例外的に手動承認できる運用を検討する。
- package manager の lockfile を必ず利用する。

### JavaScript / TypeScript

対象:

- Next.js
- React
- TypeScript
- UI ライブラリ
- ESLint / Prettier などの開発依存

方針:

- `package-lock.json` / `pnpm-lock.yaml` / `yarn.lock` のいずれかを必ずコミットする。
- `npm install` ではなく、CI では `npm ci` または lockfile を尊重するコマンドを使う。
- パッケージ取得は Takumi Guard 経由にする。
- `latest` 指定を避ける。
- 不要な依存を増やさない。
- メンテナンスされていないパッケージを避ける。

### Python

対象:

- FastAPI
- SQLAlchemy
- Pydantic
- Scraper 関連ライブラリ
- LLM API client
- Batch 処理用ライブラリ

方針:

- `requirements.txt` / `uv.lock` / `poetry.lock` などの lockfile を利用する。
- パッケージ取得は Takumi Guard 経由にする。
- `latest` 相当の無制限アップデートを避ける。
- 本番環境では lockfile に基づいて再現可能なインストールを行う。
- 依存パッケージの脆弱性スキャンを CI に組み込む。

## SQL インジェクション対策

FastAPI から PostgreSQL にアクセスする際は、SQL インジェクション対策を徹底する。

方針:

- 文字列連結で SQL を組み立てない。
- ORM または query builder のパラメータバインディングを使う。
- ユーザー入力を直接 SQL に埋め込まない。
- `ORDER BY` や `sort` など、パラメータ化しづらい値は allowlist で検証する。
- `party_id`, `category_ids`, `sort`, `limit`, `cursor` などの query parameter は Pydantic で検証する。
- DB ユーザーには必要最小限の権限のみ付与する。

例:

```txt
NG: f"SELECT * FROM articles WHERE title = '{title}'"
OK: SQLAlchemy / parameter binding を利用する
```

## XSS 対策

Next.js で外部ソース由来のテキストや LLM 出力を表示するため、XSS 対策が必要である。

方針:

- LLM 出力やスクレイピング結果を HTML として直接描画しない。
- `dangerouslySetInnerHTML` を原則使わない。
- Markdown を表示する場合は sanitize する。
- 外部サイト由来のタイトル・本文・要約はテキストとして扱う。
- URL は scheme allowlist で検証する。
- `javascript:` URL を禁止する。

## CSRF 対策

ログインや Cookie ベースのセッションを使わないため、一般的な認証付き CSRF リスクは低い。

ただし、匿名集計 API への不要な投稿を防ぐため、以下を検討する。

- CORS の許可 origin を制限する。
- `POST /analytics/*` に rate limit を設ける。
- 不正な payload は Pydantic で拒否する。
- 異常な連続投稿を検知する。

## CORS 設定

FastAPI の CORS は最小限にする。

方針:

- 本番では許可 origin を Kuni-Musubi のフロントエンドドメインに限定する。
- `*` を本番で使わない。
- 許可 method を必要最小限にする。
- 許可 header を必要最小限にする。

## Rate Limit

匿名 API であっても、過剰アクセスやスパム投稿を防ぐ必要がある。

対象:

- `GET /articles`
- `GET /articles/{article_id}`
- `POST /analytics/article-event`

方針:

- IP ベースまたは edge 側で rate limit を設定する。
- analytics API は短時間の連続投稿を制限する。
- スクレイピング対策として、異常なアクセスパターンを検知する。

## 入力バリデーション

API に渡される値はすべて検証する。

対象:

- `party_id`
- `category_ids`
- `sort`
- `limit`
- `cursor`
- `event_type`
- `surface`

方針:

- Pydantic schema で型と値を検証する。
- `sort`, `event_type`, `surface` は allowlist にする。
- `limit` には上限を設ける。
- 不明な ID は 404 または空結果として扱う。

## localStorage の扱い

localStorage には、ユーザーの表示設定のみ保存する。

保存するもの:

- 年代
- 支持政党
- 関心テーマ
- オンボーディング完了状態

保存しないもの:

- 氏名
- メールアドレス
- 電話番号
- 住所
- ユーザー ID
- アクセストークン
- 記事保存情報
- 個人別閲覧履歴

localStorage の値はユーザーが改ざんできるため、サーバー側では信用しない。

## 匿名集計データの扱い

DB に保存する分析データは、匿名集計に限定する。

方針:

- オンボーディングと設定画面の回答は localStorage にのみ保存し、サーバーには送信しない。
- 政治思想と閲覧履歴を個人単位で結合しない。
- IP アドレスを長期保存しない。
- 公開いいね数やランキングに使わない。
- 「参考になった」はプロダクト改善用の非公開指標として扱う。

## LLM セキュリティ

LLM は記事カード・記事詳細用の表示コンテンツ生成に利用する。

方針:

- LLM はユーザーリクエスト時に実行しない。
- バッチで事前生成する。
- LLM 出力を JSON schema で検証する。
- LLM 出力を HTML として表示しない。
- 出典にない事実を追加しないようプロンプトで制約する。
- `needs_human_review` フラグを持たせ、人間確認が必要な記事を分ける。
- 政治的にセンシティブな話題は人間レビューを検討する。

## スクレイピング時の注意

政党公式サイトやニュースサイトを取得する際は、法的・技術的な配慮が必要である。

方針:

- robots.txt や利用規約を確認する。
- 過度な頻度でアクセスしない。
- User-Agent を適切に設定する。
- 取得に失敗した場合のリトライ回数を制限する。
- 取得した HTML をそのままフロントエンドに表示しない。
- 外部 URL は検証して保存する。

## Secret 管理

API key や DB password はリポジトリに含めない。

対象:

- DB 接続情報
- LLM API key
- News API key
- Takumi Guard の接続情報
- Cloud provider credentials

方針:

- `.env` を Git 管理しない。
- 本番 secret は secret manager で管理する。
- CI/CD では GitHub Secrets などを使う。
- ログに secret を出力しない。

## ログ設計

ログは障害調査に必要だが、センシティブ情報を含めない。

方針:

- request body を丸ごとログに出さない。
- localStorage の内容をログに出さない。
- 支持政党や関心テーマを個人識別子と一緒にログに出さない。
- LLM 入出力を保存する場合は、出典・用途・保存期間を明確にする。

## セキュリティチェック

開発・CI で実施するチェック:

- 依存パッケージの脆弱性スキャン
- lockfile の差分確認
- secret の混入検知
- TypeScript / Python の lint
- 型チェック
- SQL 文字列連結のレビュー
- CORS 設定の確認
- rate limit 設定の確認
