# 記事詳細画面表示設計

作成者：泉知成  
作成日：2026年4月30日  
更新日：2026年4月30日

## 目的

記事詳細画面では、ホームの記事カードで興味を持ったユーザーが、ニュースの内容をもう少し深く理解できるようにする。

Kuni-Musubi はポジティブな政治ニュースを扱うが、政党広報やプロパガンダに見えないよう、出典、一次情報、残る課題、世論・与野党からの評価をあわせて表示する。

## 表示項目

記事詳細画面に表示する基本項目は以下とする。

- タイトル
- 関連政党
- カテゴリ
- 何が良いニュースなのか
- 我々の生活への影響
- 残る課題
- 世論・与野党からの評価
- 出典・一次情報リンク
- 参考になったフィードバック

## 画面構成

### 1. タイトル

記事カードと同じタイトルを表示する。

タイトルは、ポジティブで読みたくなる表現にする。ただし、記事詳細画面では内容の検証可能性を担保するため、出典や残る課題も同じ画面内で確認できるようにする。

### 2. タグ

タイトル付近にタグを表示する。

表示するタグ:

- 関連政党
- カテゴリ

関連政党は略称で表示してよい。

カテゴリは、記事カードより多く表示してよい。ただし、タグが多すぎると読みにくくなるため、最大 4 個程度を目安にする。

### 3. 何が良いニュースなのか

このニュースのポジティブな点を説明する。

目的:

- ユーザーが「なぜこのニュースが良いニュースとして扱われているのか」を理解できるようにする。
- 単なる政党の発表ではなく、社会課題に対する具体的な前進を示す。

表示方針:

- 平易な言葉で書く。
- 政策名だけで終わらせない。
- 何が改善されたのか、何が前に進んだのかを説明する。
- 事実以上に称賛しない。

### 4. 我々の生活への影響

政策や取り組みが、ユーザーの生活にどう関係するのかを説明する。

目的:

- 政治を「遠い話」ではなく、自分の生活と関係するものとして理解してもらう。
- 税金、手取り、物価、医療、雇用、安全保障などの生活接点を明確にする。

表示方針:

- 主語はできるだけ生活者視点にする。
- 「何が変わる可能性があるのか」を短く示す。
- 過度に断定せず、必要に応じて「可能性がある」と表現する。

### 5. 残る課題

ポジティブなニュースであっても、残る課題を必ず表示する。

目的:

- プロパガンダに見えるリスクを下げる。
- ユーザーが批判的思考を失わずにニュースを読めるようにする。
- 政策の実現可能性や副作用を理解できるようにする。

表示する観点:

- 財源
- 実現可能性
- 制度設計上の課題
- 副作用
- 対象から漏れる人
- 将来世代への影響

### 6. 世論・与野党からの評価

残る課題とあわせて、世論や与野党からの評価を表示する。

目的:

- 一つの政党の発表だけで完結させない。
- 複数視点を提示し、検証可能性を高める。
- 対立を煽るのではなく、建設的な論点として見せる。

表示方針:

- 攻撃的な言葉をそのまま載せない。
- 「批判」ではなく「検討すべき論点」として整理する。
- 与党・野党・専門家・世論の視点を可能な範囲で分ける。

### 7. 出典・一次情報リンク

記事詳細画面の下部に、出典と一次情報リンクを表示する。

表示する情報:

- 一次情報リンク
- 政党公式サイト
- 政府・自治体発表
- 参考にしたニュース記事
- 取得日または公開日

一次情報が存在する場合は、できる限り一次情報を優先して表示する。

### 8. 参考になったフィードバック

記事詳細画面の下部に、匿名・非公開の「参考になった」フィードバックを配置する。

目的:

- どのようなニュース記事がユーザーにとって参考になったのかを把握する。
- 記事カードや記事詳細の情報設計を改善する。
- 公開いいね数やコメント欄のような、政治的な反応数競争を避ける。

表示例:

```txt
この記事は参考になりましたか？
[参考になった]
```

方針:

- 反応数はユーザーに公開しない。
- 政党別ランキングや人気記事ランキングには使わない。
- ユーザー ID と結合しない。
- 支持政党や年代と個人単位で結合しない。
- 匿名の記事イベントとして集計する。

「いいね」ではなく「参考になった」と表現する。  
「いいね」は政党や政策への賛同に見えやすいが、「参考になった」は情報価値への反応として扱いやすい。

## 理解支援機能

チームみらいが作成している「みらい議会」では、「ふりがな」や「説明をもっと詳しく」といった、政治情報を理解しやすくするための機能がある。

Kuni-Musubi でも、政治に詳しくないユーザーの心理的ハードルを下げるため、同様の理解支援機能は相性が良い。

ただし、MVP では必須にしない。記事詳細画面の基本構成を検証した後、必要に応じて導入する。

### ふりがな

政治用語、制度名、難読語にふりがなを表示する機能。

期待効果:

- 政治ニュースへの苦手意識を下げる。
- 若年層や政治初心者でも読みやすくなる。
- 難しい言葉で置いていかれる感覚を減らせる。

### 説明をもっと詳しく

記事内の説明を、ユーザーが必要なときだけ詳しく読める機能。

期待効果:

- 初期表示は短く保てる。
- 詳しく知りたいユーザーだけ追加説明を開ける。
- 長文化による離脱を防ぎながら、理解を深められる。

表示例:

```txt
説明をもっと詳しく
```

MVP 後に実装する場合は、LLM で生成した補足説明を事前に保存し、ユーザー操作時に表示する。表示時にリアルタイムで LLM を呼び出す設計にはしない。

## MVP で優先しないこと

- 長文の解説記事化
- SNS シェア導線の強調
- 記事保存
- 関連ニュースの表示
- AI チャットによる追加質問
- 公開いいね数の表示
- コメント欄
- ふりがなの自動表示
- 「説明をもっと詳しく」の本実装

## 今後検討すること

- 「残る課題」と「世論・与野党からの評価」を同じセクションにするか分けるか
- 出典リンクの表示形式
- 生活への影響をカテゴリ別に出し分けるか
- 関連ニュースを表示する場合の件数
- 記事詳細画面で推薦理由を表示するか
- 参考になったフィードバックの配置
- ふりがなを付ける対象語
- 「説明をもっと詳しく」で表示する補足説明の粒度
